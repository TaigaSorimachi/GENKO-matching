仕様書の更新時に現構成での実現可否を確認する際の動作手順：
- `read` を使用して、更新された仕様書と @rules/tech-stack.md を確認する。
- 以下の移行トリガーに該当する要件がないか確認する：
    1. APIで10秒以上かかる処理（重い集計、外部API連鎖、ファイル変換等）
    2. WebSocket / リアルタイム通信（Supabase Realtimeでは不足する場合）
    3. バックグラウンドジョブ（メール送信キュー、定期バッチ、非同期処理）
    4. ビジネスロジックの複雑化（DI/Module構造が有効な規模）
- 該当しない場合は何もせず終了する（過剰提案しない）。
- 該当する場合、ユーザーに該当要件と現構成の制約を具体的に示し、NestJS + Railway ($5/月) 構成への移行を提案する。
- ユーザーの承認を得てから `rules/tech-stack.md` を更新し、移行タスクを `.claude/tasks/` に作成する。
