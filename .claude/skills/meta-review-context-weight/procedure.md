ルールやスキルの追加・更新後に、コンテキスト量が Claude の精度に悪影響を与えないか検証する。

## 対象ファイル

毎回コンテキストに読み込まれるファイルのみを対象とする：
- `.claude/CLAUDE.md`
- `.claude/rules/` 配下の全 `.md` ファイル
- `.claude/skills/*/SKILL.md`（各スキルのエントリポイント）

対象外（オンデマンド読み込みのため計測不要）：
- `.claude/skills/*/procedure.md`

## レビュー手順

1. 上記の対象ファイルを `read` する。
2. 以下の3つの観点でレビューする。

### 観点1: 量の計測と肥大化チェック

対象ファイルの行数を計測し、以下の表を作成する:

| ファイル | 行数 | 判定 |
|---------|------|------|
| rules/frontend/architecture.md | XX | OK / 要圧縮 |
| ... | ... | ... |
| **合計** | **XX** | |

判定基準:
- 1ファイル100行超 → 「要圧縮」（分割または削減を検討）
- 全ファイル合計500行超 → 「全体量が多い」と警告

### 観点2: ファイル間の重複チェック

異なるファイルに同じ趣旨の記述がないか確認する。

チェック項目:
- 同じ技術的指示が複数ファイルに書かれていないか（例: 命名規則がarchitectureとtestingの両方にある）
- 同じ禁止事項が複数箇所で言い換えられていないか
- スキルがルールの内容をそのまま繰り返していないか

重複を見つけた場合:
- 1箇所に正規化し、他は参照（「@rules/xxx.md に従う」）に置き換える修正案を提示する

### 観点3: 暗黙知の重複チェック

ルールに明記しなくても Claude が標準的に行う振る舞いが、わざわざ記述されていないか確認する。

削除候補の例:
- 「TypeScript で書く」→ tech-stack.md で TypeScript と指定済みなら自明
- 「コードは読みやすく書く」→ 主観的かつ Claude のデフォルト行為
- 「適切なエラーハンドリングをする」→ 具体性がなければ書いても効果なし
- 「テストを書いてからコミットする」→ CLAUDE.md のプロセスで制御すべき事項

判定基準:
- **削除すべき**: Claude のデフォルト動作と同じ or 具体性がなく行動を変えない記述
- **残すべき**: Claude のデフォルトと異なる指示、または具体的なパターン・形式の指定

## レビュー結果の報告

### サマリ
- 総行数と肥大化の有無
- 重複箇所の数
- 暗黙知の重複箇所の数

### 詳細

| ファイル | 観点 | 該当箇所 | 問題 | 修正案 |
|---------|------|---------|------|--------|
| ... | 肥大化 / 重複 / 暗黙知 | ... | ... | ... |

問題がなければ「コンテキスト量に問題なし」と報告する。
問題がある場合はユーザーの承認を得てから修正する。
