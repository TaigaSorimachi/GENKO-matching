# バックエンドテストルール

@rules/testing-common.md の共通ルールに従う。以下はバックエンド固有のルール。

## テスト戦略

統合テストで正常系とエラーレスポンスをカバーする。

### 統合テスト（メイン）
- Handler関数（GET, POST等）を直接呼び出し、Service → Repository → テストDBを通して一連のフローを検証する（HTTPサーバーは起動しない）
- 正常系を優先し、バリデーションエラー（400）、認証エラー（401/403）、Not Found（404）もカバーする

### ユニットテスト（補完）
- 統合テストではカバーしづらい複雑なロジックに限定する
- 対象: 条件分岐が多いServiceのビジネスロジック、複雑なドメインエラーの判定
- Repository層はモックし、Service層のロジックのみを検証する

### Validator
- `test.each` で正常入力・異常入力をテーブル駆動でまとめる
- 全フィールドを個別にテストせず、代表的なパターンに絞る

## テストDB管理
- テスト用のDBスキーマは `prisma migrate` で本番と同一にする
- 各テストファイルの実行前にトランザクションを開始し、終了後にロールバックする
- テスト間のデータ汚染を防ぐため、テスト同士はデータを共有しない

## テストファイルの配置

```
__tests__/                              ← プロジェクトルート直下に配置
  integration/
    [resource].test.ts              ← 統合テスト（エンドポイント単位）
  unit/
    [resource].service.test.ts      ← ユニットテスト（必要な場合のみ）
  factories/
    [resource].factory.ts           ← テストデータファクトリ
```
