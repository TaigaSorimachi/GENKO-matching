# テスト共通ルール

フロントエンド・バックエンド共通のテストルール。各領域固有のルールは `frontend/testing.md`、`backend/testing.md` を参照。

## 基本方針
- behavior（何が起きるか）を検証し、implementation detail（どう実装したか）は検証しない
- Arrange-Act-Assert パターンで統一する
- フレームワークやライブラリの動作はテストしない

## テストコードの書き方

### 構造・サイズ
- `describe` で対象をグルーピングし、`it` で具体的な振る舞いを記述する
- ネストは最大2階層
- 1テストファイルは100行以内を目安とする。超える場合はテスト対象の分割を検討する

### テスト名
- `it("〜する")` の形式で、期待される振る舞いを日本語で書く
- NG: `it("test1")`, `it("正常系")` — 何をテストしているか不明

### テストデータ
- テストデータはファクトリ関数（`createMockUser()` 等）で生成し、各テストにベタ書きしない
- 各テストではファクトリのデフォルトから必要な差分のみ上書きする

### 類似パターンの集約
- 入力値だけが異なるテストケースは `test.each` でテーブル駆動にまとめる

## テスト戦略
- 統合テストを主軸とし、ユニットテストで複雑な分岐を補完する
- 単純なCRUDは統合テスト1ケースで十分。テストの「網羅」は目指さない

## リグレッション防止
- 既存機能を修正する場合、修正前にテストがパスすることを確認してから変更する
- バグ修正時は、そのバグを再現するテストを先に書いてから修正する（TDD）
